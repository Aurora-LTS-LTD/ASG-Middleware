# Tech Spec — Aurora v0.2.4+ : CI/CD Hardening · Dual-Backend Client · M1 ITA Cutover

> Source-of-truth blueprint for the Frontend & Backend agents. Grounded in the current repo; **reuse** the existing
> implementations called out below rather than rebuilding.

## Context
The v0.2.4 cutover split the backend into two Cloud Run services and surfaced three risks this spec hardens:
- **M2** (`services/aurora-api-core`, `app.main_core:app`) is **live on v0.2.4** with real GCP Storage + BigQuery.
- **M1** (`services/aurora-main-api`, `app.main:app`) is **held on v0.2.3** — its `backend_check` requires
  `ITA_BACKEND=production` (real Israel Tax Authority), pending credentials + compliance sign-off.
- A fresh dependency resolve **drifted the unpinned web stack** (starlette 0.52→1.2 major); and the CI **import-gate
  passed an image whose startup later 503'd**, because `import app.main` never runs the FastAPI startup lifespan.

Shared code lives in `shared_packages/aurora_shared` (built to a wheel by each Dockerfile; must NOT appear in the lock).

## PILLAR 1 — CI/CD Hardening & Dependency Management  (Backend/DevOps)

### 1.1 Dependency pinning — `requirements.txt`
Pin the web stack to the prod-verified versions with tight upper bounds, and promote `starlette`/`pydantic` to direct deps:
```
fastapi>=0.136,<0.137
uvicorn>=0.48,<0.49
starlette>=1.2,<1.3
pydantic>=2.13,<2.14
sqlalchemy>=2.0.50,<2.1
```
Leave utility libs (`httpx`, `python-multipart`, `passlib`, …) flexible. **Re-lock ONLY in a clean Linux/Python-3.14
container** (never macOS — that leaked `pyobjc`); canonical procedure in
`docs/runbooks/v0.2.4-real-backends-cutover.md` STEP 4. `requirements.txt` does NOT list `anthropic`/`webauthn`
(lock-only) — a relock must add them. The current `requirements.lock` already satisfies these pins (fastapi 0.136.3,
starlette 1.2.1, uvicorn 0.48.0, sqlalchemy 2.0.50, pydantic 2.13.4), so no immediate relock is required.

### 1.2 Lifespan boot-gate — `cloudbuild_verify.yaml` (M1) + `cloudbuild.core.yaml` (M2)
The `import-gate` (`python -c "import app.main"`) imports routers but never runs `@app.on_event("startup")`
(`app/main.py:240`, `validate_backend_selectors()` at `:260`) — why the M1 v0.2.4 candidate passed verify yet 503'd.
Add a step that runs the startup lifespan via `TestClient(app)` in a `with` block (dev mode: `AURORA_RUNTIME=`,
`DATABASE_URL=sqlite://`, `SKIP_SEED_ADMIN=1`), asserting the health endpoint returns 200, before the push step.

**Two-layer model:** the dev-mode lifespan-gate catches *startup-code* failures pre-push (the import-gate's gap); it
does NOT replicate the *prod* `backend_check` hard-fail (env-specific: ITA/STORAGE/AUDIT must be non-stub in
`cloud_run`) — that stays validated at the no-traffic candidate boot with real prod env.

### 1.3 Verification
Run `gcloud builds submit . --config=cloudbuild_verify.yaml --substitutions=_IMG=…/api:<tag>` → `import-gate` **and**
`lifespan-gate` green before push. Same for M2 via `cloudbuild.core.yaml`.

## PILLAR 2 — Dual-Backend Client (Dual-Context → Multi-Window, phased)

### Reuse foundation (already built — do NOT rebuild)
- `front-end/accountant-portal/src/lib/api/client.ts`: `M1_TAX_URL`/`M2_CORE_URL` (env `NEXT_PUBLIC_AURORA_API_BASE` /
  `NEXT_PUBLIC_AURORA_CORE_BASE`), `call(path,{engine:"m1"|"m2"})`, `baseForEngine()`, routing table, single-JWT
  keychain auth + single-flight refresh. **One JWT works on BOTH backends** (shared DB + JWT secret) — by design.
- `src/components/cockpit/Cockpit.tsx`: dual-pane `M1Panel | M2Panel` (split/toggle) + `ViewSwitcher`/`EngineStatusChip`.
- `src/lib/cockpit/context.tsx` (`ENGINES`, `useCockpit`) + `useEngineHealth` (per-engine `/health` probes).
- `src/lib/tauri/keychain.ts` + `src-tauri/src/lib.rs`: OS-native, process-independent token store (shared across windows).

### Phase 1 — Engine-bound clients (dual-context isolation, one window)
`engine` is a per-call opt-in param today → a call site can cross-contaminate. Make isolation **structural**:
- **Engine-bound clients:** `createEngineClient(engine)` exporting `apiM1`/`apiM2` whose base URL is fixed at
  construction; call sites use `apiM1.get(path)` / `apiM2.post(path,body)` and cannot target the other engine. Split
  `api.*` into M1 and M2 surfaces (keep the existing auth/refresh internals).
- **Config layer:** one module exporting `M1_API_BASE_URL` + `M2_API_BASE_URL` (from the existing env vars).
- **State isolation:** namespace React Query keys by engine (`["m1",…]`/`["m2",…]`); per-pane error boundaries.
- **CSP:** add the M2 core domain to `connect-src` in `src-tauri/tauri.conf.json` (currently only `api-aurora-lts.com`).

### Phase 2 — True multi-window (separate OS windows) + state-sync bridge
Only one Tauri window is defined today. Define `m1` (Tax/Billing/Compliance) + `m2` (AI/Core/Native) windows; each
mounts the app scoped to one engine. Make `CockpitProvider` per-window. **State-sync bridge** for auth/lock — tokens
are already shared via the OS keychain; for live signals use Tauri events / `BroadcastChannel` carrying **state-only**
messages, never API payloads:
```ts
type WindowBridgeMsg =
  | { type: "auth.signed_out" } | { type: "auth.refreshed"; at: number }
  | { type: "app.locked" | "app.unlocked" } | { type: "session.expired" };
```

### No-cross-contamination guarantee
1. Each engine-bound client has a fixed base URL — no call can switch engines.
2. The shared JWT is intentional (one identity, two backends), not contamination.
3. The state-sync bridge carries only local state signals, never API requests.
4. Health probes and RQ caches are per-engine.

## PILLAR 3 — M1 ITA Cutover Readiness  (DevOps, gated on compliance)

**Gate:** `app/config/backend_check.py` `_HARD_FAIL` = `ITA_BACKEND∈{mock,stub,""}`, `STORAGE_BACKEND∈{stub,""}`,
`AUDIT_BIGQUERY_BACKEND∈{stub,""}`. Storage+audit are solved; **ITA is the remaining M1 blocker**, and enabling it
means M1 issues **real Israel Tax Authority allocations** (`app/services/ita/{client,auth,vat_filing}.py`).

### Env-var + secret payload (inject once business clears the regulatory blocker)
**Plain env** (`gcloud run services update aurora-api --update-env-vars=…`):

| Var | Value | Notes |
|---|---|---|
| `ITA_BACKEND` | `production` | **HARD-FAIL** today; currently `mock` |
| `ITA_SOFTWARE_HOUSE_ID` | `<ASG's ITA registration id>` | **Required, currently MISSING**; JWT `iss` |
| `ITA_API_BASE` | `https://ita.gov.il/api` | default OK |
| `ITA_ALLOCATION_PATH` | `/allocation/v1/issue` | default OK |
| `ITA_AUDIENCE` | `ita.gov.il` | JWT `aud`; default OK |
| `ITA_JWT_TTL_SECONDS` / `ITA_TIMEOUT_SECONDS` | `300` / `15` | defaults OK |
| `ITA_VAT_API_BASE` / `ITA_VAT_FILING_PATH` / `VAT_FILING_BACKEND` | defaults / `stub` | keep VAT off for v0.2.4 |
| `STORAGE_BACKEND` / `GCS_BUCKET_RECEIPTS` | `gcs` / `aurora-lts-prod-secure-storage` | already solved |
| `AUDIT_BIGQUERY_BACKEND` / `GOOGLE_CLOUD_PROJECT` / `BIGQUERY_AUDIT_DATASET` | `gcp` / `aurora-lts-prod` / `asg_audit` | already solved |

**Secret** (`--update-secrets`): `ITA_PRIVATE_KEY_SECRET=AURORA_ITA_PRIVATE_KEY:latest` — a **PEM RSA private key** read
via `get_secret()` (`SECRET_BACKEND=gcp`) and used in `app/services/ita/auth.py` to sign the **RS256 JWT**
(`iss`=software-house-id, `sub`=seller tax id, `aud`=`ITA_AUDIENCE`, `exp`=+TTL, `jti`=request id):
```bash
gcloud secrets create AURORA_ITA_PRIVATE_KEY --data-file=ita_private_key.pem --project=aurora-lts-prod
gcloud secrets add-iam-policy-binding AURORA_ITA_PRIVATE_KEY \
  --member=serviceAccount:aurora-run@aurora-lts-prod.iam.gserviceaccount.com --role=roles/secretmanager.secretAccessor
```
**Also pending (team):** create BigQuery tables `asg_audit.action_logs` + `asg_audit.ita_audit_log` (audit export 404s
without them; doesn't block boot).

### M1 cutover sequence (when cleared)
Set env+secret → build via `cloudbuild_verify.yaml` (build + import-gate + lifespan-gate) → no-traffic candidate →
boot gate (`Server is ready`, `backend_check` passes) → gated **10→50→100** with synthetic load through the public LB
→ rollback target `aurora-api-00055-reh` → tag `v0.2.4`. Full detail in `docs/runbooks/v0.2.4-real-backends-cutover.md`.
