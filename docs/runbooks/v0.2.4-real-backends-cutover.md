# Runbook — v0.2.4 Cutover: Enable Real Storage + Audit Backends

**Status:** BLOCKED on infra provisioning · Production safe on **v0.2.3** (`aurora-api-00055-reh`, 100%)
**Prepared:** 2026-06-02 · Project `aurora-lts-prod` · Region `me-west1` · Service `aurora-api` (M1)

## Why this exists
v0.2.4 introduces `app/config/backend_check.py` (new in the `ad2840f` merge; **absent in v0.2.3**). In production
(`AURORA_RUNTIME=cloud_run`) it **hard-fails startup** if `STORAGE_BACKEND` or `AUDIT_BIGQUERY_BACKEND` is `stub`.
Production currently runs **fully stubbed** (GCS writes are no-ops; audit/BigQuery events are discarded), which v0.2.3
tolerated silently. v0.2.4 refuses to perpetuate that. Enabling real backends is therefore a **first-time
provisioning** of production storage + audit, with compliance implications (KYC/tax retention, audit residency).

A canary on 2026-06-02 confirmed the failure: candidate `aurora-api-00064-pog` returned `503` on every request with
`RuntimeError: STORAGE_BACKEND='stub' / AUDIT_BIGQUERY_BACKEND='stub' is forbidden in production`. Rolled back; no
organic traffic affected.

## Current state (do not skip — preconditions)
- **Code fix already shipped:** the import regression that crashed v0.2.4 is fixed, committed (`90d2b58`), and pushed
  to `origin/feature/monorepo-restructure`. The v0.2.4 image (`fb5c39be`) is import-clean and boots in dev mode.
- **DB already migrated:** the prod DB has the full v0.2.4 schema (5 new tables + `invoices.kind`/`original_invoice_id`).
- **Verified missing (read-only audit, aurora-lts-prod):**
  - GCS buckets `asg-receipts-prod` (storage default) and `asg-kyc-prod` — **do not exist**.
  - BigQuery dataset `asg_audit` (+ tables `action_logs`, `ita_audit_log`) — **do not exist** (code does NOT auto-create).
  - SA `aurora-run@aurora-lts-prod.iam.gserviceaccount.com` — **missing** `roles/storage.objectAdmin` and `roles/bigquery.dataEditor`.
  - `requirements.lock` — **missing all 6 `google-cloud-*` SDKs** (offline wheel build needs them + transitive deps).

## ⚠️ DECISIONS the infra/compliance team must make first
1. **Bucket region & retention** — single-region `me-west1` (co-located with the service/DB) vs multi-region; retention
   / lifecycle policy for tax/KYC docs (commonly 7 years). Uniform bucket-level access (recommended).
2. **One bucket or two** — `STORAGE_BACKEND=gcs` uses the *receipts* bucket (`GCS_BUCKET_RECEIPTS`, default
   `asg-receipts-prod`). KYC signed-URL uploads are a *separate* path needing `asg-kyc-prod` **and** a SA key
   (`GCS_KYC_SA_KEY_JSON`) — only required if you enable `KYC_BACKEND=gcs` too (currently stub, WARN-only, not a blocker).
3. **BigQuery dataset location** — must satisfy data-residency requirements.
4. **IAM scope** — project-level grants (simpler) vs bucket/dataset-scoped (least privilege; recommended).

## STEP 1 — Provision GCS storage bucket
```bash
PROJECT=aurora-lts-prod
REGION=me-west1                       # DECISION: confirm region
RECEIPTS_BUCKET=asg-receipts-prod     # = GCS_BUCKET_RECEIPTS default

gcloud storage buckets create "gs://${RECEIPTS_BUCKET}" \
  --project="${PROJECT}" --location="${REGION}" \
  --uniform-bucket-level-access --public-access-prevention

# DECISION: retention (example: 7y) — apply only if compliance requires immutability
# gcloud storage buckets update "gs://${RECEIPTS_BUCKET}" --retention-period=2557d
```
(If enabling KYC GCS later, repeat for `asg-kyc-prod` and provision `GCS_KYC_SA_KEY_JSON` — out of scope for this cutover.)

## STEP 2 — Provision BigQuery audit dataset + tables
The exporter (`app/services/compliance/bigquery_export.py`) streams via `insert_rows_json` into
`<project>.<dataset>.<table>` for tables **`action_logs`** and **`ita_audit_log`**. Tables MUST pre-exist with a schema.
```bash
DATASET=asg_audit                     # = BIGQUERY_AUDIT_DATASET default
BQ_LOCATION=me-west1                  # DECISION: confirm residency

bq --location="${BQ_LOCATION}" mk --dataset "${PROJECT}:${DATASET}"
```
**Table schemas:** derive from the redacted payloads of the `ActionLog` and `ItaAuditLog` models
(`shared_packages/aurora_shared/aurora_shared/database/models.py`); `_redact_row` serializes datetimes → ISO strings.
Engineering should finalize the JSON schema, then:
```bash
bq mk --table "${PROJECT}:${DATASET}.action_logs"   ./schemas/action_logs.json
bq mk --table "${PROJECT}:${DATASET}.ita_audit_log" ./schemas/ita_audit_log.json
```
> Note: streaming inserts require a defined schema — an empty/no-schema table will reject `insert_rows_json`.

## STEP 3 — Grant IAM to the runtime service account
```bash
SA=aurora-run@aurora-lts-prod.iam.gserviceaccount.com

# Least-privilege (recommended): scope to the bucket + dataset.
gcloud storage buckets add-iam-policy-binding "gs://${RECEIPTS_BUCKET}" \
  --member="serviceAccount:${SA}" --role="roles/storage.objectAdmin"

# BigQuery dataset-scoped editor (project-level shown; dataset-scoped preferred):
gcloud projects add-iam-policy-binding "${PROJECT}" \
  --member="serviceAccount:${SA}" --role="roles/bigquery.dataEditor"
```

## STEP 4 — Re-lock dependencies + rebuild v0.2.4  ⚠️ REDO REQUIRED
> **The lock currently on `feature/monorepo-restructure` (commit `1a8268d`) is BROKEN** — it was a raw
> **macOS** `pip freeze` that (a) DROPPED curated prod deps `gunicorn`, `psycopg[binary]`, `anthropic`,
> `webauthn`, `jiter`, `cachetools`, `cbor2`, `distro`, `asn1crypto`; (b) added **macOS-only** `pyobjc-core` +
> `pyobjc-framework-*`; (c) added an `-e git+https://…#egg=aurora_shared` self-ref (removed in `2634411`).
> Building from it fails (`PyObjC requires macOS`) and would yield a non-functional image. **Regenerate it.**

**Regenerate in a clean Linux / Python-3.14 env** (NOT macOS — that's what leaked `pyobjc`). Do NOT hand-edit
(risk of a missing lazily-imported `google-cloud` transitive that passes build/boot/health and fails in prod):
```bash
# Linux container, python 3.14:
python3.14 -m venv /tmp/relock && . /tmp/relock/bin/activate
pip install --upgrade pip
pip install -r requirements.txt          # already pins the 6 google-cloud-* SDKs + app deps
pip freeze | grep -vE '^(pywebview|pyobjc|bottle|proxy_tools)' | sort > requirements.lock
# Confirm these survived (re-add pinned if any are absent): gunicorn psycopg[binary] anthropic jiter webauthn
# Confirm present: google-cloud-storage google-cloud-bigquery google-api-core google-auth googleapis-common-protos
#                  grpcio protobuf proto-plus google-crc32c google-resumable-media cachetools rsa
# Confirm ABSENT:  any '-e git+' / VCS line, and any pyobjc/pywebview/bottle/proxy_tools
git add requirements.lock && git commit -m "build(deps): regenerate Linux lock w/ google-cloud-* (replaces broken 1a8268d)" && git push origin feature/monorepo-restructure
```
A known-good base (all prod deps, no macOS junk, **no** google-cloud) is the `7d09da1` lock — useful for diffing:
`git show 7d09da1:requirements.lock`. The corrected lock = that base **+** the resolved `google-cloud-*` subtree.
Once pushed, the cutover (Phases A→B below) can run.
Then rebuild + push the v0.2.4 images **build-only** (no migrate/deploy/cutover):
```bash
# M1 (aurora-api) — use cloudbuild_verify.yaml: build + in-image `import app.main` gate + push.
#   The import-gate reproduces gunicorn worker boot, so a lock missing a router dep (anthropic/
#   webauthn/etc.) FAILS here and the image is never pushed — a strong guard against the broken lock.
gcloud builds submit . --config=cloudbuild_verify.yaml \
  --substitutions=_IMG=me-west1-docker.pkg.dev/aurora-lts-prod/aurora/api:v0.2.4 --project=aurora-lts-prod

# M2 (aurora-api-core) — build-only:
gcloud builds submit . --config=cloudbuild.core.yaml \
  --substitutions=_VERSION=v0.2.4 --project=aurora-lts-prod
```
> ⚠️ Do **NOT** use the stock `cloudbuild.yaml` for M1 — it is a **full** pipeline (build → migrate → deploy →
> **100% cutover**). `cloudbuild_verify.yaml` (M1) and `cloudbuild.core.yaml` (M2) are the build-only configs; cut
> over via the gated STEP 6 / Phase A–B process. (Note: the import-gate runs in dev mode, so it catches import/dep
> failures but NOT the prod `backend_check` — that is validated at the no-traffic candidate boot in STEP 5.)

## STEP 5 — Set env vars + deploy a no-traffic candidate
```bash
gcloud run deploy aurora-api \
  --image="me-west1-docker.pkg.dev/aurora-lts-prod/aurora/api:v0.2.4" \
  --region=me-west1 --project=aurora-lts-prod --no-traffic --tag=candidate \
  --update-env-vars=STORAGE_BACKEND=gcs,AUDIT_BIGQUERY_BACKEND=gcp,GOOGLE_CLOUD_PROJECT=aurora-lts-prod,GCS_BUCKET_RECEIPTS=asg-receipts-prod,BIGQUERY_AUDIT_DATASET=asg_audit
```
**Boot gate:** confirm the new candidate's workers actually load (not just "Ready"):
```bash
REV=$(gcloud run services describe aurora-api --region=me-west1 --format='value(status.traffic[?tag=`candidate`].revisionName)')
gcloud logging read "resource.labels.revision_name=\"$REV\" AND (textPayload:\"Server is ready\" OR severity>=ERROR)" \
  --project=aurora-lts-prod --freshness=5m --order=asc --limit=50
# PASS = "[STARTUP] Server is ready!", NO RuntimeError / "Worker failed to boot".
```

## STEP 6 — Gated traffic cutover (with active canary load)
Because organic traffic is near-zero, **drive synthetic load** so the canary is actually exercised.
```bash
CAND=<candidate-revision>     # e.g. aurora-api-006X-xxx
STABLE=aurora-api-00055-reh   # v0.2.3 rollback baseline

# 10%
gcloud run services update-traffic aurora-api --region=me-west1 \
  --to-revisions=${CAND}=10,${STABLE}=90
for i in $(seq 1 100); do curl -s -o /dev/null -w "%{http_code}\n" \
  https://api-aurora-lts.com/api/v1/onboarding/health; done | sort | uniq -c   # expect ALL 200
# inspect candidate: 0 5xx, 0 tracebacks, "Engine bound" present
# 50% (repeat load + check)  →  100%
gcloud run services update-traffic aurora-api --region=me-west1 --to-revisions=${CAND}=50,${STABLE}=50
gcloud run services update-traffic aurora-api --region=me-west1 --to-revisions=${CAND}=100
```
**Post-cutover verification:** upload a receipt/vault doc → expect `[STORAGE/gcs] uploaded` + an object in
`gs://asg-receipts-prod/...`; trigger `POST /api/v1/internal/audit-export` → rows land in `asg_audit.action_logs`.

